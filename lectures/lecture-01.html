LLM에서 Agent로

언어 모델은 답을 생성합니다. 에이전트는 목표를 해석하고, 필요한 도구를 선택하고, 결과를 관찰한 뒤 다시 생각합니다. 첫 강의는 이 전환을 하나의 이야기로 따라갑니다.

Paradigm Shift

문제는 “모델이 똑똑한가”가 아니라 “행동할 수 있는가”입니다.

사용자가 “이번 주 서울 날씨를 보고 야외 행사 공지를 작성해줘”라고 요청했다고 가정해봅시다. LLM은 문장을 만들 수 있지만, 최신 날씨를 직접 알 수는 없습니다. 에이전트는 여기서 한 단계 더 나아갑니다. 필요한 정보를 얻기 위해 도구를 부르고, 그 결과를 다시 읽어 최종 답변을 구성합니다.

LLM only 질문 -> 추론 -> 답변

훈련 시점 이후의 사실이나 외부 상태에는 약합니다.

Agent 질문 -> 추론 -> 도구 -> 관찰 -> 답변

실시간 정보와 계산 결과를 문제 해결 과정에 통합합니다.

Anatomy

에이전트는 두뇌와 몸을 분리해서 이해하면 쉽습니다.

B

Brain

LLM이 목표를 해석하고 다음 행동을 계획합니다. 시스템 메시지는 역할과 규칙을 고정합니다.

T

Tools

검색, 계산, API 호출, 파일 처리처럼 모델 밖의 능력을 함수 인터페이스로 제공합니다.

E

Environment

도구 실행 결과와 외부 상태가 관찰로 돌아오며, 다음 추론의 재료가 됩니다.

전통적 프로그램과 에이전트의 차이

규칙 기반 코드는 예측 가능성이 강점입니다. 에이전트는 예측 가능성 일부를 양보하는 대신, 불완전한 요청을 해석하고 도구를 조합하는 유연성을 얻습니다.

Story Walkthrough

하나의 요청이 에이전트 내부에서 흐르는 방식

01

목표를 읽는다

“행사 공지 작성”이라는 최종 산출물과 “이번 주 서울 날씨”라는 필요한 근거를 분리합니다.

02

도구를 고른다

실시간 정보가 필요하므로 내부 지식 대신 날씨 API 또는 웹 검색 도구를 선택합니다.

03

결과를 관찰한다

비 예보, 기온, 시간대 같은 값을 읽고 메시지의 톤과 행동 안내를 조정합니다.

04

사용자에게 답한다

근거를 포함한 최종 문구를 작성하고, 필요한 경우 우천 대안까지 제안합니다.

weather_tool.py

                    
from smolagents import tool

@tool
def get_weather(city: str) -> str:
    """
    Return the current weather summary for a city.

    Args:
        city: City name such as "Seoul" or "Paris".
    """
    return weather_api.lookup(city)

First CodeAgent

                
smolagents는 이 루프를 적은 코드로 시작하게 해줍니다.

            


            

                
agent.py

                
from smolagents import CodeAgent, DuckDuckGoSearchTool, HfApiModel

model = HfApiModel()
tools = [DuckDuckGoSearchTool()]

agent = CodeAgent(
    model=model,
    tools=tools,
)

agent.run(
    "이번 주 서울 날씨를 검색하고 야외 행사 공지를 3문장으로 작성해줘."
)
