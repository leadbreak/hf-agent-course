What is an Agent?
지능형 에이전트의 구조적 해부

언어 모델의 정적 텍스트 생성을 넘어, 자율적 추론과 환경 상호작용 능력을 갖춘 에이전트 시스템의 동작 메커니즘을 마스터합니다.

1. 에이전트의 정의와 이분법 구조 (Brain & Body)

Hugging Face Agents Course에서 정의하는 에이전트(Agent)는 사용자가 정의한 목표를 달성하기 위해 AI 모델을 활용하여 외부 환경과 능동적으로 상호작용하는 시스템입니다[cite: 18, 19]. 에이전트는 본질적으로 다음 두 가지 요소의 결합체입니다[cite: 20].

  • 두뇌 (The Brain - AI Model): 모든 추론(Reasoning)과 계획 수립(Planning)이 일어나는 핵심 엔진입니다[cite: 20, 21]. 현재의 상황을 분석하고 목표 달성을 위해 어떤 행동(Action)을 취할지 결정합니다[cite: 21]. 주로 LLM이나 VLM(Vision-Language Model)이 이 역할을 맡습니다[cite: 33, 36].
  • 신체 (The Body - Capabilities & Tools): 에이전트에게 허용된 실행 능력의 범위입니다[cite: 22]. 에이전트는 자신이 장착하고 있는 도구(Tools) 목록을 기반으로 환경에 개입하고 행동을 수행합니다[cite: 14, 23]. 날개가 없는 인간이 날 수 없듯이, 에이전트의 실행력은 부여된 도구의 한계에 종속됩니다[cite: 23, 24].

자율성 스펙트럼 (The Spectrum of Agency)

에이전트 시스템은 프로그램의 제어 흐름(Control Flow)을 LLM의 출력물이 얼마나 주도하느냐에 따라 연속적인 자율성 스펙트럼 상에 존재합니다[cite: 25].

자율성 수준 패턴 분류 (Pattern) 제어 흐름의 특징 (Control Flow) 예시 의사코드
☆☆☆ (Level 0) Simple Processor LLM 출력이 전체 프로그램 흐름에 영향을 주지 않음 [cite: 27] process_llm_output(resp) [cite: 27]
★☆☆ (Level 1) Router LLM의 판단에 따라 기본적인 분기문(If/Else) 제어 [cite: 28] if llm_decide(): path_a() [cite: 28]
★★☆ (Level 2) Tool Caller LLM이 실행할 함수와 인자(Arguments)를 직접 결정 [cite: 29] run_function(chosen_tool, args) [cite: 29]
★★★ (Level 3) Multi-step Agent LLM이 루프의 종료 및 지속 여부를 스스로 판단하며 주도 [cite: 30] while llm_should_continue(): [cite: 30]
★★★ (Level 4) Multi-Agent 하나의 에이전트 워크플로우가 다른 자율 에이전트를 트리거 [cite: 31] if llm_trigger(): execute_agent() [cite: 31]

2. LLM 엔진의 본질과 Chat Template 평면화

에이전트의 두뇌 역할을 하는 디코더 기반 LLM(Decoder-based LLM)은 본질적으로 '이전 토큰들이 주어졌을 때 다음 토큰을 예측'하는 자동회귀(Autoregressive) 모델입니다[cite: 84, 88, 92, 111]. 모델은 텍스트 입력을 받아 텍스트를 출력할 뿐이며, 자체적으로 과거를 '기억'하는 메모리 공간을 가지고 있지 않습니다[cite: 33, 159].

⚠️ Chat History Flattening (대화 기록 평면화)
사용자가 웹 UI를 통해 에이전트와 주고받는 다회차 대화(User and Assistant turn)는 개발자 전용 추상화 레이어일 뿐입니다[cite: 158]. LLM 엔진이 구동될 때마다, 과거의 모든 대화 내역 목록은 단 하나의 거대한 단일 프롬프트 시퀀스 문자열로 완전히 합쳐져서(Concatenated) 모델에 처음부터 끝까지 통째로 주입됩니다[cite: 158, 175, 176].

특수 토큰(Special Tokens)과 프로바이더별 다양성

단일 문자열로 평면화된 프롬프트 속에서 시스템 지침, 사용자의 발화, 에이전트의 응답 영역을 명확히 구분하기 위해 모델 고유의 특수 토큰(Special Tokens)이 사용됩니다[cite: 99, 101, 166]. 모델은 텍스트를 생성하다가 문장을 끝내야 할 타이밍에 문장 종료 토큰(EOS, End of Sequence)을 내뱉고 멈추게 됩니다[cite: 102, 112].

Model Engine Provider EOS Token (종료 토큰) 구조적 역할 및 기능 [cite: 103]
GPT-4 OpenAI <|endoftext|> 메시지 텍스트의 끝을 명시 [cite: 104, 105]
Llama 3 Meta AI <|eot_id|> 전체 대화 시퀀스 및 회차의 종료를 명시 [cite: 105, 106]
DeepDouble-R1 DeepSeek <|end_of_sentence|> 생성된 메시지 스트림의 결속 지점을 처리 [cite: 106]
SmolLM2 Hugging Face <|im_end|> ChatML 지침 또는 특정 인스트럭션 메시지 종료 [cite: 106, 107]
Gemma Google <end_of_turn> 단일 대화 턴(Turn)의 경계를 확정 [cite: 107, 108]

인터넷의 원시 말뭉치를 학습한 Base 모델은 단순 문장 이어쓰기만 수행합니다[cite: 179]. 이를 대화형 및 도구 호출형 시스템으로 쓰기 위해, 지시사항을 따르도록 파인튜닝한 변형이 Instruct 모델입니다[cite: 180]. 이 과정에서 프롬프트를 일관되게 구조화하기 위해 Jinja2 템플릿 엔진 기반의 transformers.apply_chat_template() 함수가 백엔드에서 필수적으로 동작합니다[cite: 188, 193, 197].

3. AI Tools 설계와 Python Introspection 자동화

도구(Tool)는 에이전트가 정적 학습 데이터의 시공간적 한계(Knowledge Cut-off)를 깨부수고 실시간 데이터에 접근할 수 있게 해주는 파이썬 함수입니다[cite: 213, 221]. 에이전트 라이브러리 내부에서 유효한 도구로 인정받기 위해서는 다음의 명세가 완벽하게 정의되어 문자열 형태로 시스템 프롬프트에 주입되어야 합니다[cite: 223, 235, 270].

  • 도구의 이름 (Name): LLM이 호출 타겟을 식별할 수 있는 명확한 명칭[cite: 223, 251, 261].
  • 텍스트 설명문 (Description): 이 도구가 어떤 기능을 수행하고 언제 써야 하는지 기술한 설명[cite: 223, 251, 262].
  • 인자 유형 및 타입 힌트 (Arguments with Typings): 입력 파라미터의 명칭과 명확한 데이터 타입[cite: 224, 251, 263].

@tool 데코레이터를 통한 자동 명세화 (Auto-formatting)

매번 이러한 텍스트 스키마를 수동으로 작성하는 것은 오류에 취약합니다[cite: 241, 245]. 현대적인 에이전트 프레임워크(예: smolagents)는 파이썬의 **인트로스펙션(Introspection, inspect 모듈)** 기능을 활용해 소스 코드의 `docstring`과 `type hint`를 파싱하여 LLM용 설명 문자열을 자동으로 빌드합니다[cite: 252, 253, 254, 267].

import inspect from typing import Any # 에이전트 프레임워크 내부의 @tool 데코레이터 메커니즘 의사코드 def tool(func: callable) -> Any: class GeneratedTool: def __init__(self): self.name = func.__name__ # 함수명이 도구 이름이 됨 [cite: 247, 261] self.description = func.__doc__ # docstring 주석이 설명문이 됨 [cite: 247, 262] self.fn = func def to_string(self) -> str: # inspect 모듈로 인자 명세와 타입을 추출하여 시스템 프롬프트용 텍스트 변환 [cite: 252, 254, 266] sig = inspect.signature(self.fn) return f"Tool Name: {self.name}\nDescription: {self.description}\nSignature: {sig}" return GeneratedTool() # 실무 사용 예시 @tool def calculate_market_share(revenue: int, total_market: int) -> float: """특정 기업의 매출액과 전체 시장 규모를 기반으로 시장 점유율(%)을 계산합니다.""" return (revenue / total_market) * 100

이렇게 생성된 정보는 시스템 프롬프트 내에 바인딩되어 LLM에게 주입되며, 최근에는 모든 프레임워크와 모델에서 범용적으로 도구를 플러그인할 수 있도록 표준화된 오픈 프로토콜인 **MCP(Model Context Protocol)** 기술이 각광받고 있습니다[cite: 270, 271].

4. ReAct 사이클과 Stop & Parse 메커니즘의 실체

단순히 단계별 논리적 흐름만 타는 Chain-of-Thought(CoT)와 달리, ReAct 아키텍처는 추론 단계 사이사이에 실제 외부 환경의 도구를 실행하고 그 결과값(Observation)을 관찰하는 행동 단계를 강제로 끼워 넣습니다[cite: 306, 310, 313, 314].

🚨 더미 에이전트 실습을 통해 본 LLM의 '환각(Hallucination)' 문제
Hugging Face Course의 더미 에이전트 빌드 실습 코드를 보면 알 수 있듯이, 모델에게 단순히 ReAct 포맷으로 텍스트를 생성하라고 명령한 뒤 끝까지 내버려 두면, 모델은 자신이 도구를 실행하지 않았음에도 불구하고 프롬프트 하단에 가상의 Observation: 날씨가 맑음 이라는 **거짓 데이터(Hallucinated Observation)**를 스스로 창작하여 계속해서 텍스트를 출력해 버리는 치명적인 결함이 발생합니다[cite: 408, 409].

Stop & Parse 아키텍처 파이프라인

에이전트 프레임워크는 이 환각 현상을 방지하고 현실 세계의 코드를 구동하기 위해 정밀한 'Stop & Parse' 제어 파이프라인을 가동합니다[cite: 340].

  1. 1단계: 의도된 포맷 생성 (Structured Generation)
    LLM이 상황을 분석한 뒤 시스템 약속에 맞춰 Action: 도구이름 및 인자 정보가 포함된 구조화된 블록(JSON 또는 코드 블록)을 생성합니다[cite: 341].
  2. 2단계: 토큰 생성 강제 중단 (Halting via Stop Sequence)
    텍스트 스트리밍 도중 도구 실행 명세가 끝나는 지점(예: Observation: 키워드 직전)에 도달하면, 백엔드 엔진이 작동하여 LLM의 **토큰 생성을 강제로 중단(Halt)**시키고 제어권을 파이썬 래퍼 프로그램으로 회수합니다[cite: 339, 342, 410].
  3. 3단계: 문자열 파싱 및 실행 (Parsing & Execution)
    외부의 전용 파서(Parser)가 모델이 출력한 문자열을 읽어 실행할 함수와 파라미터를 추출하고, 실제 로컬 환경이나 외부 API를 통해 도구를 실행합니다[cite: 343].
  4. 4단계: 컨텍스트 강제 주입 (Observation Appending)
    도구의 실제 리턴값(API 결과 로그, 에러 메시지 등)을 텍스트로 변환하여 기존 대화 시퀀스의 맨 마지막 줄에 Observation: [실제결과값] 형태로 강제 인젝션한 뒤, 업데이트된 프롬프트를 들고 LLM의 다음 턴을 호출합니다[cite: 370, 372, 384].

5. 아키텍처 대결: JSON Agent vs Code Agent

초기의 펑션 콜링(Function Calling) 시스템이나 전통적인 LangChain 에이전트들은 주로 LLM에게 엄격한 JSON 포맷으로 호출 규격을 생성하도록 유도했습니다[cite: 330, 332]. 그러나 Hugging Face 프레임워크 아키텍트들은 왜 **직접 실행 가능한 파이썬 코드를 뱉어내는 'Code Agent'**로의 대전환을 선언했을까요? [cite: 348]

자료 스니펫에서 증명하듯, Code Agent는 네이티브 인터프리터 환경을 매개로 동작하며 JSON 구조 대비 다음과 같은 기술적 압도 우위를 가집니다[cite: 331, 348].

  • 표현력과 제어 흐름의 최적화 (Expressiveness): JSON 아키텍처는 다단계 연산이나 조건 분기문이 필요할 때마다 매번 ReAct 루프를 새로 돌며 LLM 추론 API를 호출해야 하므로 토큰 비용과 오버헤드가 폭증합니다[cite: 349]. 반면, Code Agent는 LLM이 단 한 번의 추론 패스로 내부 루프(`for`, `while`)와 조건 제어문(`if/else`)을 포함한 완전한 알고리즘 스크립트를 짜서 한 번에 실행시킬 수 있습니다[cite: 349].
  • 상태 저장 및 모듈성 (Modularity): 코드 블록 내에서 변수(Variables)를 선언해 중간 실행 결과값을 유연하게 재사용할 수 있어 복잡한 데이터 파이프라인 처리에 탁월합니다[cite: 350].
  • 추적 디버깅 편의성 (Debuggability): 스키마 에러 판정이 모호한 JSON과 달리, 코드 실행 중 예외가 발생하면 파이썬 고유의 명확한 Traceback 로그 메시지를 수집하여 LLM에게 다시 피드백 컨텍스트로 제공할 수 있으므로 자가 수정(Self-Correction) 확률이 급상승합니다[cite: 351].

# Code Agent가 한 번의 루프로 처리하는 다단계 연산 파이썬 코드 블록의 실체 [cite: 356] def execute_agent_step(): # 1. 주식 데이터를 API 호출하여 가져옴 [cite: 356] raw_data = fetch_financial_api(ticker="HuggingFace") # 2. JSON 에이전트라면 별도 루프를 돌아야 하는 필터링/루프 가공을 내부 코드 샌드박스에서 즉시 완료 [cite: 349] filtered_prices = [x['close'] for x in raw_data if x['volume'] > 5000] moving_average = sum(filtered_prices) / len(filtered_prices) # 3. 최종 연산된 값 확인 [cite: 356] print(f"이동평균 계산 완료: {moving_average}") # 이 출력이 Observation 필드로 회수됨 [cite: 357] execute_agent_step()

※ 모델이 생성한 원시 코드를 그대로 실행하는 것은 보안상 위험(Prompt Injection 등)을 초래하므로, 실무 환경에서는 smolagents가 제공하는 디폴트 보안 보호 규격이나 완전하게 격리된 가상 머신 샌드박스 환경 내에서 실행하는 것이 표준 보안 프로토콜입니다[cite: 353, 354].
